# Аналитическая таблица проектов | Газпромнефть-ЦР

Привет! Представляю решение тестового задания для команды Практики бизнес и продвинутой аналитики.

Проект представляет собой SPA для работы с большими массивами данных (10 000+ записей проектов и пул из 250+ менеджеров) с упором на
производительность, правильную работу с памятью и строгую архитектуру.

## 🛠 Стек технологий

- **React 18** + **TypeScript** (Strict Mode)
- **AG Grid v33** (Enterprise / Server-Side Row Model) — для работы с большой таблицей
- **Consta UI Kit** — для UI-компонентов (Select, Button, TextField, Checkbox, Text, Badge)
- **react-virtuoso** — для виртуализации длинных списков
- **json-server** — Mock API
- **ESLint** + **Husky** + **lint-staged** — для контроля качества кода и архитектуры

## 🚀 Запуск проекта

Убедитесь, что у вас установлен Node.js (рекомендуется v18+). Husky настроен через скрипт `prepare`, поэтому хуки инициализируются
автоматически при установке зависимостей.

```bash
# 1. Установка зависимостей (автоматически инициализирует Husky)
npm install

# 2. Запуск mock-сервера (API будет доступно на localhost:3001)
npm run server

# 3. Запуск клиентского приложения (в отдельной вкладке терминала)
npm start
```

## 🏗 Архитектура и контроль качества (Killer Feature)

В проекте реализована строгая архитектура **Feature-Sliced Design (FSD)**.

Чтобы архитектура не деградировала с развитием проекта, внедрен **автоматический pre-commit контроль**:

- Используется плагин `@conarti/eslint-plugin-feature-sliced`.
- Настроен хук `husky` + `lint-staged`.
- **Как это работает:** Перед каждым коммитом линтер проверяет измененные файлы на нарушение слоев (например, чтобы фичи не импортировали
  другие фичи, а общение с модулями шло строго через Public API — `index.ts`). Если архитектурное правило нарушено — коммит блокируется.
- Виджеты (например, `FiltersPanel` и `ProjectsTable`) полностью развязаны между собой через инверсию зависимостей (TypeScript
  интерфейсы-контракты), что делает их независимыми переиспользуемыми модулями.

## 🧠 Ключевые технические решения

### 1. Умный фильтр менеджеров (Inverted Selection)

Загружать 250 менеджеров в DOM нельзя, а отправлять массив из 248 ID при снятии двух галочек — неоптимально для сети.

- **Решение:** Реализован паттерн Inverted Selection (`SelectionMode.Include` / `SelectionMode.Exclude`).
- Если пользователь нажимает "Выбрать все", режим переключается на `Exclude`. Стейт выбранных ID очищается.
- Если после этого снять 2 галочки, в стейт попадают только эти 2 ID, а на сервер уходит лаконичный запрос с оператором отрицания:
  `managerId_ne=[id1, id2]`.
- Оптимизированы циклы при массовом выборе (устранены вложенные переборы).

### 2. Серверная пагинация (AG Grid SSRM)

Таблица не рендерит 10 000 записей сразу.

- Используется `ServerSideRowModelModule` от AG Grid. Данные запрашиваются чанками (`cacheBlockSize`).
- Фильтрация и сортировка (по Бюджету и Прогрессу) делегируются на сервер.
- Настроены кастомные Cell Renderers: прогресс-бары с цветовой индикацией порогов (80/40), бейджи статусов (`Consta UI`), форматирование
  валюты с разделителями через `Intl.NumberFormat`.

### 3. Производительность и защита от Race Conditions

- **Виртуализация:** Список менеджеров рендерится через `react-virtuoso`. При скролле срабатывает подгрузка (Infinite Scroll) без просадок
  FPS.
- **AbortController:** При быстром вводе текста в поиск менеджеров предыдущие pending-запросы отменяются. Это защищает от состояния гонки
  (race conditions) и экономит ресурсы браузера.
- **Оптимизация рендеров:** Устранен классический баг с двойным вызовом API при инициализации списка (за счет функционального обновления
  стейта `prev => ...` и правильной настройки массива зависимостей в `useCallback`).

## 🛠 Технический долг (Что можно улучшить)

Проект готов к масштабированию, но в рамках боевой разработки я бы предложил следующие доработки:

1. **Глобальная обработка ошибок (Error Boundaries & Toasts):** Сейчас ошибки запросов перехватываются и логируются в консоль. Необходимо
   добавить `ErrorBoundary` для изоляции падений UI и систему нотификаций (Toasts от Consta) для информирования пользователя.
2. **State Management / Caching:** Для более сложной работы с API имеет смысл переехать с нативных `fetch` на FSD-совместимый слой работы с
   сетью (например, RTK Query или TanStack Query) для кэширования и автоматической дедупликации запросов.
3. **Скелетная загрузка (Skeleton):** Заменить базовые спиннеры (`Loader`) на скелетоны в таблице и списке фильтров для более плавного UX.
4. **Итоги по отфильтрованным данным:** Добавить отдельный запрос агрегации (или endpoint на бэкенде) для вывода суммарного бюджета и
   среднего прогресса по текущим фильтрам над таблицей.
